    • sass main.scss main.css
  • Automatische Konvertierung
    • sass --watch main.scss:main.css

    • sass --watch "Ordner"
  • Styling Optionen
    • sass --style nested/expanded/compact/compressed

SASS vs. SCSS


  • SASS benötigt keine ; { }
  • SASS muss immer korrekt eingerückt werden (auch Leerzeichen)
  • SCSS (Sassy CSS) ist einfacher zu verstehen, sehr an CSS angelehnt



Beispiel Sassmeister

Unterschiede zwischen SCSS und CSS


  • Dateiendungen .sass und .scss aktuell nicht von Browsern lesbar
  • CSS ist leichter zu verstehen
  • Bereits kompilierte Dateien im Cache
  • // Kommentare nur in SASS bzw. SCSS lesbar
  • /* */ Kommentar auch im CSS Code sichtbar
  • SASS/SCSS Code besser geeignet für größere Projekte mit vielen wiederholenden Elementen

Nesting



  • Verschachtelte Selektoren
  • Domänen definieren
  • angesprochenes Element genauer spezifizieren
  • Code wird lesbarer und individueller
  • Problem: viele Selektoren werden wiederholt
  • DRY (Don't repeat yourself)
  • Inception Rule (nicht weiter als 4 Ebenen gehen)

@import



Syntax: @import "Dateiname";

  • erweitert CSS @import um SASS und SCSS Dateien
  • SASS und SCSS Dateien mit Unterstrich im Namen nicht in CSS Code importiert
  • zu viele Style-Dateien verursachen unnötige HTTP Requests
  • SASS kann kleinere Style Dateien in eine große Style Datei verschmelzen

@extend



  • Code sollte semantisch sein (blue_button != checkout_button)
  • fasst Selektoren mit gleichen Attributen zusammen
  • Attribute können überschrieben werden
  • Placeholder Selektor (%) macht @extend unsichtbar im CSS

@mixin

Mixin anlegen


@mixin MinxinName(Argumente) {}

"@" + "mixin" + "MixinName" (+ "Argumente") + "{Inhalt}"

  • Funktion ähnlich zu @extend
  • Attribute und Werte können nicht überschrieben werden
  • Argumente optional
  • Default Werte direkt in Argumenten

@mixin

Mixin verwenden


@include MinxinName(Argumente);

"@" + "MixinName" (+ "Argumente") + ";"
