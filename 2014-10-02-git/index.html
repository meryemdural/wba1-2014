1 Was ist Git? Und warum nutzen wir es?

  • Ein modernes Versionskontrollsystem, entwickelt von Linus Torvalds
  • Verwaltung von zeilenbasierten Änderungen an Dateien
  • Dezentrale „Repositories“

  • Vorteile von Git:

    • Optimiert auf gemeinsames, verteiltes Arbeiten
    • Besonders gute Toolunterstützung / 3rd-Party-Apps

2 Was ist GitHub? Und warum nutzen wir es?

  • Eine zentrale Infrastruktur für Git (eine „Autorität“)

    ▸ „ein zentrales Repository“

  • Von der OpenSource-Community und Arbeitgebern akzeptiert

    ▸ „Social Development“

  • GitHub bietet zudem optional:

    • Webinterface für alle wichtigen Funktionen
    • Task-Verwaltung „Issues“
    • Ein Wiki pro Projekt
    • Eine Webseite pro Projekt
    • Hosting...

3 Grundbegriffe

Was ist ein Repository?

Eine zusammengehörige Menge von Dateien die in einer "Git-Instanz" verwaltet werden

  • init → Lokales Repository anlegen
  • clone → Entferntes Repository kopieren

3 Grundbegriffe

Was ist ein Repository?

Was ist ein Commit?

Eine zusammengehörige Menge von Änderungen an einem Repository

  • status oder „changes“ → Änderungen anzeigen
  • commit → Änderungen versionieren

Wie werden Daten synchonisiert?

Die Synchronisierung über erfolgt über eine zentrale Instanz (z.B. Github)

  • pull → Änderungen runterladen und in lokale Kopie „mergen“
  • push → Änderungen an entferntes Repository senden

Was ist ein Branch?

Eine Menge von Commits, die vom "Master-Branch" abgezweigt werden

  • branch → Ermöglicht paralleles arbeiten an unterschiedlichen „Features“
  • merge → Bezeichnet das zusammenführen unterschiedlicher branches

4 Konflikte

  • Wann treten Konflikte auf?
  • → Bei Änderungen, die nicht automatisch mergebar sind.
  • → Beim synchonisieren (pull und mergen)
  • Kann man das Risko reduzieren?
  • → Häufig(er) synchonisieren
  • → Mit „Umbauarbeiten“ (etwa neu formatieren) zurückhalten - ergo kleinere Pakete häufiger syncronisieren

5 GitHub und WBA1

  • Der Standardbranch heißt normalerweise „master“, bei uns aber „gh-pages“
  • Unsere Konventionen für die Arbeit am Repository:
    • Wir verwenden branches, Schema bitte: html-einstieg-a
    • Ordner bitte nach dem Schema: 2013-12-06-html-einstieg-a
  • Wir möchten GitHub Pull Requests für das mergen verwenden
  • Wir möchten GitHub Issues für die Kommunikation (TODOs, etc.) verwenden
